# Glossary

A working glossary of the concepts, tools, and terms that show up across the exercises and
hypotheses in this repository. It follows the arc of *High Performance Python* (Gorelick &
Ozsvald): how to **measure** a program (Chapter 2), how Python's built-in **containers** are
actually laid out in memory (Chapters 3–4), how **lazy iteration** changes the memory story
(Chapter 5), and how **numpy and the GPU** move work out of the interpreter and onto the
hardware (Chapter 6). Definitions describe each term as the exercises use it, not in the abstract.

Cross-cutting ideas (big-O, cache locality, amortization) are collected at the end since they
recur in every chapter.

---

## Profiling & measurement (Chapter 2)

The chapter's running example is the **Julia set**, a fractal generated by the escape-time
algorithm, used as a deliberately CPU-bound workload to profile from every angle.

| Term | Definition |
| --- | --- |
| **Julia set** | A fractal computed by iterating `z = z*z + c` for a fixed complex constant `c`, coloring each starting point by how many iterations it survives before escaping. The chapter's benchmark workload — pure CPU-bound arithmetic with a verifiable output (the 1000×1000 grid at 300 iterations always sums to `33219980`, the book's fixture checksum). |
| **Escape-time algorithm** | Repeatedly apply `z = z*z + c` and count iterations until `abs(z)` exceeds 2 (the point "escaped") or a max count is hit. The inner `while abs(z) < 2 and n < maxiter` is the hot loop, run ~34 million times for the default grid. |
| **CPU-bound** | A workload limited by raw computation, not by waiting on disk or network. The Julia set runs at ~99% single-core utilization, so nearly all its time is in code you can optimize. |
| **Wall-clock time** | Elapsed real-world time, start to finish. Contrast **user time** (CPU in your code) and **system time** (kernel/OS calls); for a CPU-bound job, wall-clock ≈ user time and system time is negligible. |
| **Interpreter overhead ("interpreter tax")** | The fixed per-operation cost CPython pays to run bytecode: instruction dispatch, type checks, refcount updates. A single `n += 1` compiles to five bytecode ops (`LOAD_FAST`, `LOAD_SMALL_INT`, `BINARY_OP`, `STORE_FAST`, `JUMP_BACKWARD`) — why pure-Python loops are slow and vectorization helps. |
| **`timeit`** | Python's microbenchmarking module. Auto-calibrates the loop count and reports the **best of N** runs (the minimum), since noise can only ever *add* time. |
| **`@timefn`** | A small custom decorator in this chapter that wraps a function and prints its wall-clock time, separating e.g. grid-construction cost from the escape-time loop without a full profiler. |
| **Profiler** | A tool that measures where a program spends time or memory. The chapter works through `cProfile`, `line_profiler`, `memory_profiler`, `py-spy`, `scalene`, `memray`, and `viztracer`, each trading granularity against overhead. |
| **Deterministic (instrumenting) profiler** | Records *every* call or line by injecting measurement into the execution path — accurate but slow (`line_profiler` can inflate runtime ~10×). `cProfile` and `line_profiler` are of this kind. |
| **Sampling profiler** | Periodically peeks at the call stack (~100 Hz) instead of instrumenting every call, so overhead is near zero and it can attach to a running process. `py-spy` and `scalene` work this way. |
| **`cProfile`** | Python's built-in deterministic profiler. Records every call's timing into a `.prof` stats file, inspected with `pstats`, sorted by **tottime** or **cumtime**. |
| **Total time (tottime)** | Time in a function's *own* body, excluding callees. Sort by this to find the direct hotspot. |
| **Cumulative time (cumtime)** | Time in a function *plus everything it calls*. High cumtime with low tottime marks a "coordinator" frame that orchestrates work but isn't the bottleneck. |
| **`line_profiler` / `kernprof`** | Deterministic profiler that breaks timing down per source *line*, with hit counts. Mark functions with `@profile`, run via `kernprof -l -v`. |
| **`memory_profiler` / `mprof`** | Memory rather than time. `memory_profiler` reports **RSS** before/after each line (its **increment** column shows per-line growth) and is slow; `mprof` samples RSS as a low-overhead time-series, with labeled **phases** via `profile.timestamp("label")`. |
| **RSS (Resident Set Size) / Max RSS** | Physical RAM the process is actually using; "max RSS" is its peak. The headline memory number from `memory_profiler`, `mprof`, and `/usr/bin/time`. |
| **`py-spy`** | A sampling profiler needing no code changes; can attach to a running process by PID. Its **%Own** mirrors cProfile's tottime, **%Total** mirrors cumtime, and it reports GIL contention. |
| **`scalene`** | A low-overhead sampling profiler that splits each line's time into **Python** (interpreter) vs **native** (C) work — e.g. revealing `abs()` spends its time in CPython's C code — and captures CPU and memory in one run. |
| **`memray`** | Bloomberg's allocator-level profiler. Intercepts every `malloc`/`calloc`/`mmap` and flame-graphs the **allocation sites** — the right tool for memory leaks, and (unlike older WSGI middleware) works with async/ASGI apps. |
| **`viztracer` / `vizviewer`** | A tracer recording the entry/exit timestamp of every call, rendered as an interactive, zoomable **timeline** (Perfetto-style) viewed with `vizviewer`. Shows the *sequence* and nesting of calls over time. |
| **`snakeviz`** | A browser viewer for `cProfile` output, drawing an interactive **icicle graph** (a flame graph oriented downward). |
| **`gprof2dot` + Graphviz** | Turns a `.prof` file into a **call graph** PNG — nodes are functions, edges show who calls whom, and a reddish "spine" marks the hot path to the hottest leaf. |
| **`flameprof`** | Converts a `cProfile` stats file into a static SVG **flame graph**. |
| **Flame graph** | Stacks call frames vertically (height = call depth) and sizes bars horizontally by time (width = cumulative time). Wide bars near the bottom are where the program lives. Interactive versions (d3, py-spy SVG) zoom and search; static ones (`flameprof`) are images. |
| **GIL (Global Interpreter Lock)** | CPython's lock letting only one thread execute bytecode at a time. For the single-threaded Julia set, `py-spy` reports "GIL: 100%"; the GIL is why threading doesn't speed up pure-Python compute. |
| **Memory leak** | Objects allocated but never freed, so RSS climbs until the OS **OOM-kills** the process (exit code 137 = 128 + SIGKILL). A true leak shows a **monotonically rising** object count; the `leaky_*.py` demos reproduce one in a web service. |
| **Boxed object** | A heap-allocated Python wrapper around a single value. A complex number costs ~40 bytes (value + type pointer + refcount + the list slot pointing at it) — the memory penalty that motivates numpy's unboxed arrays. |

## Lists & tuples (Chapter 3)

How Python's two core sequence types are laid out, and the searching that sorted sequences unlock.

| Term | Definition |
| --- | --- |
| **Dynamic array** | The structure behind a `list`: a contiguous memory block that grows by allocating a bigger block and copying everything over when full. O(1) indexing, at the cost of occasional expensive resizes. |
| **Contiguous memory / uniform stride** | Elements packed sequentially at a constant size, so element `i` lives at `base + i * stride`. What makes list indexing O(1) and gives lists their good **cache locality**. |
| **Overallocation** | On growth, CPython reserves *more* slots than needed (formula `M = (N + (N>>3) + 6) & ~3`, ~+12.5% headroom). Keeps reallocations logarithmic and `append` **amortized O(1)**, at the cost of dead **headroom**. |
| **Amortized O(1) append** | An individual `append` that triggers a resize is O(n), but those rare resizes spread thin across the many cheap appends between them, so the *average* cost per append is constant. |
| **Preallocation** | Building a list in one shot (`[None] * N` or `list(range(N))`) instead of repeated `append`. Avoids the repeated resize-and-copy and produces an exact-sized list with no slack. |
| **Resize bookkeeping** | The hidden metadata (~8 bytes) a list keeps to track allocated capacity for growth. Immutable tuples omit this word — part of why tuples are leaner. |
| **Tuple immutability** | A tuple's contents are fixed at creation, which lets it drop resize bookkeeping and lets CPython safely recycle small tuples via a freelist. |
| **Tuple freelist** | CPython caches pre-allocated empty tuple objects per small size (0–20 elements, up to ~2,000 each). Creating a small tuple reuses one instead of calling the allocator — an order-of-magnitude speedup. |
| **Freelist cliff** | The sharp discontinuity at size 20→21, where tuples fall off the freelist and must be freshly allocated. The h01 hypothesis measures it as a ~16 ns jump in creation time. |
| **Binary search** | An O(log n) search over a *sorted* sequence that halves the remaining range each comparison. The basis for everything in `bisect`. |
| **`bisect`** | Stdlib binary search on sorted lists: `bisect_left` / `bisect_right` find an insertion index in O(log n); `insort` inserts while keeping order (O(n), since insertion still shifts elements). |
| **Linear search** | An O(n) element-by-element scan — what `list.index` and `in` do on a list. The **worst case** is an absent value, forcing a full traversal. |
| **Membership testing** | Checking `x in seq`. On a list, a linear O(n) scan whose cost depends on where (or whether) the item is found — the motivation for sets and dicts. |
| **Total ordering** | Defining how custom objects compare so they can be sorted or binary-searched. Supply `__eq__` and `__lt__`; `functools.total_ordering` derives the other four operators. |

## Dictionaries & sets (Chapter 4)

The hash table — the structure under both `dict` and `set` — and the consequences of hashing for
speed and memory.

| Term | Definition |
| --- | --- |
| **Hash table** | The structure behind `dict` and `set`. Stores entries in **buckets** indexed by a number derived from each key's hash, giving average-case O(1) lookup, insert, and delete. |
| **Hash function (`__hash__`)** | A deterministic key→integer map. A *good* one spreads keys uniformly; a **degenerate** one (e.g. a constant) collapses everything into one bucket and degrades the table to a linear scan. |
| **Hash mask** | How a full hash folds to a bucket index: `hash(key) & (size - 1)`. Tables are powers of two, so `size - 1` is a clean run of low bits and the fold is a single bitwise AND. |
| **Power-of-two sizing** | Tables grow through 8, 16, 32, 64, … so the mask is a contiguous block of low bits. The ex09 exercise shows that for integers (`hash(n) == n`) collisions come *entirely* from the mask discarding high bits, not a weak hash. |
| **Collision** | When two distinct keys map to the same bucket. Unavoidable in general, so the table needs a strategy to keep both. |
| **Open addressing & probe sequence** | If a key's home bucket is taken, CPython follows a deterministic **probe sequence** to a free slot. The sequence is **perturbed** — mixing in high-order hash bits — to avoid the clustering of naive linear probing. The ex04 exercise traces it step by step. |
| **Tombstone** | A marker left in a deleted bucket meaning "empty, but something used to be here," so an in-progress probe doesn't stop early and miss a key that probed past this slot. |
| **Load factor** | How full the table gets before growing: ≤2/3 for dicts, ≤3/5 for sets. Keeping it sparse keeps probe chains short and lookups near O(1). |
| **Resizing** | When the load factor is exceeded, CPython allocates a bigger table and re-inserts every key with fresh indices. Triggering inserts are O(n), but cost amortizes to O(1). |
| **Insertion sawtooth** | Visible spikes in per-insert cost at each power-of-two boundary, where a resize rehashes everything. The h01 hypothesis plots it; the teeth grow taller as the table grows. |
| **Set vs list uniqueness** | Deduplicating with a `set` is O(1) per element (a hash probe) → O(n) overall; with a list and `in` it's O(n) per element → O(n²). The ex02 exercise measures the gap. |
| **Hashable** | Usable as a dict key or set member: needs a stable `__hash__`, and if it overrides `__eq__` the two must agree (equal objects hash equal). The ex05 `Point` shows **content-based** hash/equality (value-equal points collapse to one entry) vs the default **identity hash** (memory address), under which they stay distinct. |
| **Hash entropy** | How evenly a hash spreads keys. The h02 hypothesis shows lookup cost degrading *smoothly* from O(1) toward O(n) as entropy drops — no sharp cliff, which is why a mediocre hash ships unnoticed. |
| **Hash randomization (hash seed)** | CPython salts string hashing with a per-process random seed, so which keys collide changes run to run. Defends against hash-collision DoS attacks. |
| **String hash caching** | A string computes its hash once (O(length)) and stores it inside the object, so every later hash of the same string is O(1). The h04 hypothesis confirms it. |
| **EAFP ("easier to ask forgiveness than permission")** | The idiom of attempting `try: d[k]` and catching the miss. One lookup on the hot path, but pays exception-handling cost when the key is absent — so it wins when keys are usually present. |
| **LBYL ("look before you leap")** | The idiom of guarding with `if k in d: d[k]`. No exceptions, but pays for two lookups (the check, then the access) — so it wins when misses are common. |
| **EAFP/LBYL three-way crossover** | The h03 hypothesis benchmarks EAFP, LBYL, and `dict.get()` across hit rates and finds no single winner: each idiom is fastest in a different regime, trading exception overhead against the double-lookup cost. |
| **`__slots__`** | A class declaration that fixes allowed attribute names and removes the per-instance `__dict__`, saving ~40 bytes per object for small value classes. The h05 hypothesis measures the savings. |

## Iterators & generators (Chapter 5)

Lazy evaluation: computing values on demand so memory stays flat no matter how large the stream.

| Term | Definition |
| --- | --- |
| **Iterator / iterable** | An **iterator** has `__iter__` and `__next__`, yielding one value per `next()` until **StopIteration**. An **iterable** (list, range, generator) is anything that can hand you an iterator via `iter()`. |
| **Iterator protocol** | What a `for` loop desugars to: call `iter()` once, then `next()` repeatedly until `StopIteration`. The ex01 exercise deconstructs a `for` loop into exactly these calls. |
| **Generator** | A function with `yield` (or a parenthesized generator expression) producing values on demand. A generator *is* its own iterator; each `yield` suspends it with local state intact, resumed on the next `next()`. |
| **`yield from`** | Delegates iteration to a sub-generator, forwarding values at the C level. The h01 hypothesis measures it ~13% faster than manual `for x in sub: yield x`. |
| **Lazy vs eager evaluation** | **Lazy** computes each value only when demanded, streaming single-pass with O(1) memory. **Eager** computes and stores everything up front — O(n) memory, but random access and repeat iteration. The chapter trades eager lists for lazy generators. |
| **Materialization** | Forcing a lazy iterator into a concrete collection (e.g. `list(map(...))`). Allocates and holds every value at once, discarding the memory advantage of laziness. |
| **The len trap** | The ex03 anti-pattern: `len([x for x in data if cond])` materializes the whole filtered list just to count it. `sum(1 for x in data if cond)` counts lazily in O(1) memory. |
| **Single-pass exhaustion** | A generator can only be consumed once; no rewind. Re-running the pipeline means rebuilding it — the price of laziness. |
| **Generator pipeline** | A chain of generator stages, each pulling from the one before. **Demand propagates** backward: one `next()` at the end pulls just enough work through every upstream stage to make one value, with no intermediate collections. The ex06 anomaly pipeline is the worked example. |
| **Infinite sequence** | A generator on `while True: yield ...` that never ends on its own (ex04 infinite Fibonacci). Safe because the *consumer* decides when to stop — via `islice`, `takewhile`, or `break` (**early termination**). |
| **`itertools`** | The stdlib toolkit of lazy, composable building blocks. The ex05 drills cover `islice` (take N), `chain` (concatenate streams), `cycle` (repeat forever), `takewhile` (yield until a condition fails), and `count` (count up forever, usually bounded by `islice`). |
| **Lazy vs eager built-ins** | The ex09 catalog of which built-ins stream vs materialize. **Lazy**: `map`, `zip`, `filter`, `enumerate`, `range`, `reversed`, `dict.items`. **Eager**: `list`, `sorted`. Tells you where memory is silently allocated. |
| **Rolling (sliding) window** | A fixed-size window of the last *K* items sweeping a stream (ex07). The tension: a **deque** read in place stays O(1) per step, but **rebuilding a tuple** to yield each step adds copy overhead — the right choice depends on whether the consumer uses the window immediately. |
| **Generator depth** | The number of stacked generator layers. The h02 hypothesis finds per-item CPU cost grows *linearly* with depth (~11.5 ns per layer, one extra **generator frame** to resume per item) while memory stays flat — you pay in time, not space. |
| **Class-based iterator vs generator machinery** | A hand-written `__iter__`/`__next__` class is ~2.8× slower per item than an equivalent generator (h03), because it pays Python-level method-dispatch and attribute-access on every item, whereas CPython implements generator suspend/resume in C. |

## NumPy, vectorization & hardware (Chapter 6)

Moving work out of the interpreter — into numpy's C kernels, then onto the GPU — using a 2D
diffusion simulation as the running example.

| Term | Definition |
| --- | --- |
| **Vectorization** | Replacing per-element Python loops with whole-array operations that run in compiled C (and SIMD under the hood). The single biggest lever for escaping the interpreter tax. |
| **SIMD / vector instructions** | "Single Instruction, Multiple Data": CPU instructions (SSE, AVX, NEON) applying one op to several values at once — the mechanism behind fast numpy. Per-element conversion and **horizontal reduction** (summing across a register) "vectorize less well" than straight element-wise work. |
| **numpy** | Python's array library: contiguous, unboxed numeric arrays plus compiled C kernels for bulk operations. The ex01 exercise turns a slow list-of-floats norm into a single fast `dot`. |
| **Broadcasting** | numpy's rule for operating on arrays of different shapes by automatically stretching the smaller dimensions, avoiding explicit loops. |
| **Contiguous vs strided arrays** | **Contiguous** stores elements back-to-back (vectorizes well). **Strided** (e.g. `a[::2]`) steps through memory at a fixed **stride** > 1, hurting cache behavior and blocking vectorization. `np.ascontiguousarray` copies a strided view into a fresh buffer — an allocation for better access. |
| **In-place vs out-of-place** | `a += b` writes back into `a`'s buffer (**in-place**, no allocation); `a = a + b` allocates a new array (**out-of-place**, paying allocation + page faults). The ex03 exercise measures the gap; in-place wins past the **break-even point** of buffer reuse. |
| **Temporary array** | An intermediate numpy allocates mid-expression, e.g. `v*v` in `sum(v*v)`. Eliminating these is **fusion** — collapsing an expression into one pass, as `np.dot` does for `sum(v*v)` and `numexpr` for arbitrary expressions. |
| **Diffusion equation / stencil computation** | The chapter's example: a 2D PDE solved by repeatedly updating each grid cell from its four neighbors. Any computation where a cell's new value depends on neighbors is a **stencil**. |
| **Laplacian** | The stencil at the heart of diffusion: for each cell, sum the differences to its four neighbors. Implemented here with four `np.roll` calls. |
| **`np.roll`** | Circularly shifts an array along an axis (wrapping at the edge). Convenient for stencils, but each call allocates a **temporary** — four rolls per step. The ex05 `roll_add` exercise tries to hand-specialize those away, a cautionary tale about premature specialization. |
| **Lazy allocation & page faults** | On Unix, `np.empty` reserves address space instantly without touching it, while `np.zeros` writes zeros to every page, paying a **page fault** (kernel trap mapping physical memory) on **first touch**. The h2 hypothesis and ex04 use this to explain why "allocating" can look free until you write. |
| **Cache locality / cache miss / L3 cache** | Accessing memory so cache lines get reused stays fast; an access to data not in L1/L2/L3 stalls on a **cache miss** fetch from RAM. Once an array spills past last-level (**L3**) cache, the performance pattern changes — the **cache crossover** the ex11/h1 work probes. |
| **Memory-bandwidth bound vs compute bound** | **Element-wise** `a + b` touches each element once (O(n) on O(n) data), limited by data movement — cores don't help. **Matrix multiply** `a @ b` is O(n³) on O(n²) data, compute-limited, scaling with cores. The h6 hypothesis contrasts how threading helps one but not the other. |
| **von Neumann bottleneck** | The underlying reason: memory bandwidth is fundamentally slower than CPU compute, so data movement, not arithmetic, is often the limit. |
| **Axis reduction locality** | A **reduction** (summing along an axis) has a different access pattern by direction: a **vertical sum** (axis=0, down columns) vs a **horizontal sum** (axis=1, across rows) walk memory and vectorize differently. The h1 hypothesis measures the asymmetry. |
| **numexpr** | Compiles a whole array expression (a string) into one optimized, multithreaded, cache-tiled kernel with no temporaries. Its fixed **compile/setup overhead** means it only pays off above a crossover size — the ex11 exercise and h3 hypothesis. |
| **scipy** | SciPy's scientific routines, e.g. `scipy.ndimage.laplace`. The ex12 cautionary exercise shows the general filter is *slower* than specialized diffusion code: its **generality** (many boundary modes, dtypes) forces runtime branches that block tight vectorization — **branch-free, specialized code** wins the narrow case. |
| **GPU** | A massively parallel accelerator with thousands of slow cores built for throughput, not latency. Great for **embarrassingly parallel** work, terrible for a **sequential walk** where each step depends on the last. |
| **MPS (Metal Performance Shaders)** | Apple's GPU compute backend, reached via PyTorch on Apple Silicon (the book uses CUDA on NVIDIA). The ex07/ex09 exercises run the diffusion stencil on MPS. |
| **Unified memory** | On Apple Silicon, CPU and GPU share physical RAM, so moving data between them is far cheaper than copying across NVIDIA's discrete VRAM. |
| **Asynchronous execution & synchronization** | GPU calls return immediately after *queuing*; you must `synchronize()` before timing. Reading a single GPU tensor element forces an implicit **CPU↔GPU sync** that wrecks performance. A **warm-up** run pays the one-time kernel **compile/launch cost** outside the timed region. |
| **Conv2d / convolution** | A stencil as a convolution kernel. PyTorch's hardware-tuned `Conv2d` with **circular padding** (wrap-around, matching `np.roll`) computes the same Laplacian faster than a hand-built roll version. The ex09 exercise demonstrates it. |
| **GPU saturation & batching** | A GPU only pays off once work fills its cores. **Batching** groups independent items into one submission so idle cores get used; the h4 hypothesis shows per-item cost collapsing as batch size grows, until the GPU **saturates** and per-item cost flattens. |
| **Floating-point precision (float16/32/64, bfloat16)** | Bit-width trades accuracy for memory bandwidth. **float32** is the native default; **float64** doubles accuracy and size; **float16** is GPU-friendly but needs conversion on CPUs lacking native support; **bfloat16** keeps float32's exponent range with a smaller mantissa (why DL prefers it). The ex06 exercise probes precision-vs-speed on CPU. |
| **AMP (Automatic Mixed Precision) / `autocast`** | PyTorch's `torch.autocast` context that auto-runs eligible ops (matmul, convolution) in lower precision for speed without manual casting. The ex10 exercise benchmarks bfloat16 AMP. |

## Compiling to C (Chapter 8)

Replacing the interpreter with machine code for the same CPU-bound Julia loop — four compilers
and a foreign-function bridge — measured into the book's Table 8-1 and Table 8-2.

| Term | Definition |
| --- | --- |
| **Compiling to C** | Turning Python into native machine code so the CPU executes fewer, cheaper instructions — the next lever after good algorithms and less data. Helps math-heavy, loop-heavy code; does little for I/O-bound or already-vectorized numpy. |
| **AOT (ahead-of-time) compilation** | Compile to a machine-specific binary *before* running, so it's instantly fast on first use (Cython). The trade is more up-front manual effort. |
| **JIT (just-in-time) compilation** | Compile *during* execution, specialized to the argument types actually seen (Numba, PyPy). Little up-front effort, but a **cold start** — the first call pays the compile cost, paid fresh in every new process, so JITs suit long-lived programs over short, frequently relaunched scripts. |
| **Cython** | An AOT compiler that converts type-annotated Python (a `.pyx` hybrid language) into a compiled extension module. The chapter's main tool; powers SciPy, scikit-learn, lxml. Built via `setup.py`/`cythonize` or auto-compiled by **pyximport** on import. |
| **`cdef` / type annotation** | Cython's keyword for declaring C types (`cdef unsigned int i`, `cdef double complex z`). Replacing heap Python objects with register-resident C types in the *hot inner loop* is where the speed comes from — ~15× in ex02; annotations outside the inner loop buy little. |
| **Cython annotation report (`cython -a`)** | An HTML view shading each line by how much it calls back into the Python VM (yellow = more VM, white = pure C). A targeting tool: only shaded lines inside tight inner loops are worth fixing. |
| **`memoryview` / buffer protocol** | A typed view (`double complex[:]`) over any object exposing the buffer interface — numpy arrays, `array.array`. Indexing compiles to a direct C memory offset with no VM call, which is what unlocks both C-speed dereferences and `nogil` parallelism (ex03). |
| **Strength reduction** | Rewriting an operation as a cheaper equivalent — e.g. `abs(z) < 2` → `z.real²+z.imag² < 4`, dropping a `sqrt`. **Engine-dependent**: a win once compiled, but a ~1.9× *loss* in pure CPython where `abs` is one builtin call and the expansion is many bytecodes (ex01). The compiler won't do it for you because it changes IEEE-754 semantics. |
| **OpenMP** | A cross-platform API for compiler-level shared-memory parallelism in C/C++/Fortran. Cython reaches it with `prange`; the threads live in the generated C, hidden from Python. |
| **`prange` / `nogil`** | Cython's parallel `range`, run inside a `with nogil:` block that releases the GIL. Safe only when the loop body touches C primitives and memoryviews, never Python objects. Schedules — **`static`** (even split), **`dynamic`**/**`guided`** (shrinking runtime chunks) — matter when per-item work is uneven; `guided` suits the lumpy Julia loop (ex03). |
| **Numba** | A JIT specialized for numpy code, compiling decorated functions via **LLVM** at call time. One `@jit(nopython=True)` decorator, no type annotations — warm speed ties hand-tuned Cython (ex04). `@jit(parallel=True)` enables `prange`, but exposes no OpenMP schedule control, so it trails Cython on uneven work. |
| **PyPy** | A whole alternative Python interpreter with a tracing JIT — a drop-in `python` replacement, not an importable library. ~6× on pure Python for free, but ~6× *slower* on numpy because C-extension calls cross the **cpyext** compatibility layer it can't trace through. Uses mark-and-sweep GC, not refcounting. |
| **FFI (foreign function interface)** | A bridge to call code compiled in another language. A *doorway*, not an engine: the speed comes from the C kernel, so the FFI choice is about ergonomics and safety, not performance (ex05). |
| **`ctypes`** | The stdlib FFI. `CDLL` loads a `.so`; you hand-declare `argtypes`/`restype` and cast every pointer. No type metadata in the binary to check against, so a wrong cast fails silently or segfaults. |
| **`cffi`** | A friendlier FFI (from the PyPy authors) that parses C signatures you give it (`ffi.cdef("void evolve(double**, ...)")`) and generates the marshalling — the same text as the library's header. Partial-struct support (`...;`) insulates against library version churn. |
| **`f2py`** | A numpy tool that wraps Fortran as a Python extension; Fortran's explicit types let it auto-generate the interface and hide vector sizes (`!f2py intent(hide)`). Needs a Fortran compiler (`gfortran`) and **column-major** (`order='F'`) arrays. |
| **CPython C extension** | Writing a module directly against the CPython API — maximally portable and minutely fastest, but dozens of lines of manual refcounting and argument parsing per function. The chapter's "last resort." **PyO3** (Rust) is the modern, memory-safe counterpoint. |

## Asynchronous I/O (Chapter 9)

| Term | Definition |
| --- | --- |
| **I/O-bound** | A program whose runtime is dominated by *waiting* for input/output (network, disk, database) rather than by computation. The opposite of CPU-bound, and the only regime where async I/O helps (ex01, ex05). |
| **I/O wait** | The time a program is paused waiting for the kernel to complete a read/write on a device. The CPU is idle but the program cannot proceed — ~77% of the serial CPU+I/O workload (ex05). The reclaimable resource the whole chapter targets. |
| **Concurrency vs. parallelism** | Concurrency interleaves many tasks on **one** thread, sharing resources by filling each task's dead time with another's work; parallelism (Chapter 10) gives each task its **own** resources to run literally simultaneously. Async I/O is concurrency. |
| **Event loop** | The scheduler at the heart of async code: a list of ready coroutines that it runs one at a time, switching between them whenever one yields control at an `await`. `asyncio.run(coro)` starts a temporary one (ex01–ex08). |
| **Coroutine** | A function defined with `async def`. Implemented like a generator — calling it builds an object and runs **none** of its body; it advances only when the event loop drives it with `send()` (ex04). |
| **`await`** | Suspends the current coroutine until the awaited thing is ready, handing control back to the event loop so other coroutines can run. Functionally a `yield` that the loop resumes when data arrives. |
| **`Future` / `Task`** | A promise of a result that isn't ready yet. `tg.create_task(coro)` schedules a coroutine as a `Task`; it does not run on creation, only when the loop next gets a turn (ex04). |
| **`TaskGroup`** | A 3.11+ context manager that collects tasks and, on exit (`__aexit__`, itself an `await`), guarantees they all finish or are cancelled. That exit is often the *first* yield in a creation loop, which is when the queued tasks finally run (ex02, ex04). |
| **`aiohttp` / `ClientSession`** | The async HTTP library used here; `ClientSession` pools and caches connections (defaulting to 100 simultaneous), so requests run in waves of that size (ex02). `httpx` is the HTTP/2-capable alternative. |
| **`asyncio.gather` vs. `TaskGroup`** | Two ways to run many tasks. `gather` returns results in order and (by default) keeps going if one fails; `TaskGroup` gives structured, fail-fast cancellation. ex06 uses `gather`, ex02/ex07 use `TaskGroup`. |
| **`await asyncio.sleep(0)`** | A yield that doesn't sleep: it schedules a zero-delay wakeup and suspends, forcing one trip through the event loop so pending I/O can advance. Essential in CPU-bound loops that otherwise never `await` (ex07, ex08). |
| **Cooperative scheduling** | asyncio switches tasks only at explicit `await` points (never preemptively), so code between two `await`s is race-free without locks — but a CPU loop that never yields starves every other task (ex08). |
| **Pipelining / batching** | Queuing results and flushing them in concurrent bursts instead of one at a time. Amortizes the per-request delay across the whole batch for most of the async win with little code change (ex06). |
| **`eager_task_factory`** | An opt-in factory (`loop.set_task_factory`) that runs a new task's body immediately on creation, up to its first real `await` — useful when a task often has no async work to do (e.g. a cache hit) (ex04). |
| **Connection limit** | The cap on simultaneous in-flight requests (client- or server-side). Sets the wave size and the concurrency sweet spot (~100–250); past it, event-loop dispatch becomes the bottleneck and, for fast requests, runtime *reverses* (ex03). |

## Cross-cutting ideas

These recur in every chapter, so they're collected here rather than repeated.

| Term | Definition |
| --- | --- |
| **Big-O / algorithmic complexity** | How runtime or memory scales with input size *n*: O(1) constant, O(log n) binary search, O(n) linear scan, O(n²) list-based dedup, O(n³) matrix multiply. Big-O describes *scaling* and is blind to constant factors — which is why constant-factor work like vectorization needs *measurement*, not just complexity analysis, to justify. |
| **Amortized cost** | Averaging an occasional expensive operation (a list resize, a dict rehash) across the many cheap ones around it, so the per-operation average stays low despite individual spikes. Seen in list `append`, dict insertion, and the **sawtooth** patterns at resize boundaries. |
| **Cache locality** | Accessing memory in contiguous, predictable patterns so the CPU reuses cache lines instead of stalling on RAM. Why contiguous lists and numpy arrays beat scattered, pointer-chasing layouts — and the lever behind several Chapter 6 results. |
| **Hypothesis-driven benchmarking** | The discipline structuring every `hypothesis/` folder: state a *falsifiable* prediction from a mechanism (hardware, allocator, algorithm), measure it, record a verdict. Its companion is that results are **hardware-specific** — cache sizes, instruction sets, and CPython version shift the numbers — so every measurement here is reported against CPython 3.14 on Apple Silicon, reproducible within noise on the same machine. |
